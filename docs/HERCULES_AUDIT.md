# Hercules Prototype Audit

## Purpose

This document is a reference audit of the old Hercules prototype.

It is **not** the source of truth for the new build.

The main source of truth remains:
- `docs/PRODUCT_SPEC.md`

This audit exists only to help identify:
- useful concepts worth keeping
- technical choices that should not be carried over blindly
- UI ideas that may inspire the new product
- risks, gaps, and inconsistencies in the old prototype

---

## 1. What Was Inspected

Two archives were reviewed:

1. a Convex snapshot export containing table data
2. a Vite + React + Convex prototype codebase

The old prototype appears to be an internal lead management tool with:
- authentication
- a dashboard
- a leads list
- a lead detail page
- an agencies page

It does **not** fully implement the target product scope described in `docs/PRODUCT_SPEC.md`.

## 4. Current Workflow Logic In The Prototype

### Lead intake

- A lead can be created manually from the leads page.
- A lead can also be created from an inbound webhook endpoint.
- New leads start with status `new`.

### Lead pipeline

The prototype uses this lead status model:

- `new`
- `to_qualify`
- `qualified`
- `assigned`
- `quote_in_progress`
- `quote_sent`
- `negotiation`
- `won`
- `lost`
- `archived`

### Qualification

- Qualification is handled in a modal on the lead detail page.
- It stores qualification notes.
- It can store a qualification score.
- It automatically pushes the lead to `qualified`.

### Agency assignment

- A lead can be assigned to one agency.
- Assigning an agency automatically sets the lead status to `assigned`.

### Quote creation

- Quotes are created from the lead detail page.
- Quote versions are generated by counting existing quotes for that lead.
- Creating a quote automatically sets the lead to `quote_in_progress`.
- Sending a quote updates the lead to `quote_sent`.

### Activity timeline

- Status changes create activity entries.
- Note additions create activity entries.
- Quote creation and quote sending create activity entries.
- Agency assignment creates activity entries.

### Closure

- A lead can be marked `won` or `lost`.
- A close reason can be stored.

## 5. Useful Concepts To Keep

These are conceptual ideas worth keeping in the new build, as long as they are reimplemented cleanly and aligned with `docs/PRODUCT_SPEC.md`.

### 5.1 Internal orchestration model

Useful concept:
- the tool is clearly internal-facing
- staff operate the workflow
- the traveler is not navigating an agency-facing interface

Why it matters:
- this fits the core rule in `docs/PRODUCT_SPEC.md` that Direction l’Algérie remains the single interface

### 5.2 Lead-centered workflow

Useful concept:
- most operations revolve around a lead record
- qualification, assignment, notes, and quote activity all attach to the lead

Why it matters:
- this aligns well with the new product, which is fundamentally lead-driven

### 5.3 Activity history / traceability

Useful concept:
- every important action leaves an activity entry

Why it matters:
- this is strongly aligned with the product spec rule that everything must be traceable

### 5.4 Agency directory as an internal asset

Useful concept:
- agencies are modeled as internal partners with specialties, zones, and internal scoring

Why it matters:
- this is compatible with the future `Agencies` and `Agency Consultations` modules

### 5.5 Quote versioning idea

Useful concept:
- quotes can have versions linked to a lead

Why it matters:
- the new product spec explicitly needs quote versions

### 5.6 Web-to-internal lead capture

Useful concept:
- inbound travel requests can enter the internal tool through a controlled endpoint

Why it matters:
- this supports the product vision of capturing travel requests before internal qualification

## 6. Technical Parts To Discard

These should not be migrated directly into the new app.

### 6.1 Full technical stack

Discard as implementation reference:
- Vite app structure
- React Router routing model
- Convex-specific backend and auth coupling
- Hercules auth provider setup

Reason:
- the new build already targets Next.js App Router and a different future backend direction from `docs/PRODUCT_SPEC.md`

### 6.2 Prototype package complexity

Discard:
- the very large dependency surface
- the broad collection of UI primitives added before product depth existed

Reason:
- it adds complexity too early
- it does not fit the beginner-friendly, step-by-step build approach

### 6.3 Modal-heavy workflow as the default pattern

Discard conceptually as a default:
- qualification only in a modal
- assignment only in a modal
- quote creation only in a modal

Reason:
- the future product likely needs richer dedicated workflows and clearer auditability

### 6.4 Overloading lead status as the only workflow engine

Discard as-is:
- too much business state is compressed into one status field

Reason:
- the new product spec distinguishes lead workflow, evolving need, consultations, quote versions, and internal activities more clearly

### 6.5 Direct technical migration of Convex schema

Discard:
- table names
- field naming
- mutation patterns
- backend APIs

Reason:
- the schema is useful as inspiration only, not as the new data contract

## 8. Risks Or Inconsistencies In The Old Prototype

### 8.1 Product scope is too narrow

Risk:
- the prototype covers only a subset of the target product

Examples:
- no real quotes module page
- no users admin page
- no settings page
- no consultations module
- no tasks module
- no lead snapshots module

### 8.2 The core non-negotiable rule is not modeled strongly enough

Risk:
- the prototype does not deeply encode the rule that the traveler must never communicate directly with the agency

Why this matters:
- the new product spec treats this as non-negotiable
- the old prototype only hints at internal operation but does not structurally enforce this separation

### 8.3 Quotes are under-modeled

Risk:
- quote support is present but still thin

Gaps:
- no dedicated quote workspace
- no clear traveler version vs internal version separation
- no explicit handling of internal agency notes versus client-visible output

This conflicts with:
- the quote rules in `docs/PRODUCT_SPEC.md`

### 8.4 No true consultation layer

Risk:
- assigning an agency is treated almost as a direct operational step

Missing concept:
- consultation lifecycle
- agency response tracking
- internal comparison between agency options

This is a major gap relative to:
- the `Agency Consultations` module in the new spec

### 8.5 No evolving requirement snapshots

Risk:
- the prototype stores the current lead state only

Missing concept:
- versions of the traveler need
- history of refinements
- explicit snapshots over time

This conflicts with:
- the `Evolution du besoin` section in the new spec

### 8.6 Workflow naming mismatch

Risk:
- the prototype statuses only partially match the desired future workflow

Examples:
- `to_qualify` exists in prototype
- `affinage` is a first-class concept in the new spec but not modeled clearly
- `co-construction` is central in the new spec but missing as a true stage

### 8.7 Users and roles are too shallow

Risk:
- user roles exist only minimally

Gaps:
- no real user management screen
- no meaningful permission model in UI
- role handling is thin compared with the future admin / operator / agency needs

### 8.8 Prototype naming and branding drift

Risk:
- the old prototype uses names like `TravelFlow`
- the new product must stay faithful to `Direction l’Algérie — Travel Lead Desk`

Why it matters:
- branding drift often leaks into UX and technical naming if copied without review

### 8.9 Data export shows very limited real usage

Risk:
- the snapshot contains only a very small amount of example data

Why it matters:
- it is enough to reveal structure ideas
- it is not enough to validate the workflow depth or edge cases

## 9. Recommended Migration Position

The correct way to use Hercules is:

- keep it as a conceptual reference only
- reuse only selected product ideas
- do not port its code
- do not port its architecture
- do not port its backend assumptions

Use `docs/PRODUCT_SPEC.md` as the source of truth for:
- product scope
- module boundaries
- business rules
- workflow design
- future data modeling

Use this audit only to answer:
- what was already explored before
- what seemed useful conceptually
- what should be avoided in the rebuild

---

## 10. Practical Takeaways For The New Build

### Good conceptual seeds to carry forward

- lead-centric internal workspace
- internal partner directory
- activity timeline
- quote versioning concept
- dashboard overview

### Things to redesign from scratch

- auth flow
- app architecture
- backend/data model
- consultations workflow
- quote visibility rules
- evolving lead need snapshots
- user and role system

### Safe conclusion

The old Hercules prototype is useful as:
- a rough product sketch
- a source of vocabulary
- a source of UI inspiration
- a warning about where the new product needs stronger structure

It is **not** sufficient as:
- a system design reference
- a data model reference
- a workflow source of truth
- an implementation base for the new app
